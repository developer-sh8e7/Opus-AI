# Research: CLAUDE-FABLE-5.md

## Research Question
How does the Opus AI Discord bot route user messages through deterministic handlers, AI tool execution, and memory/entity continuity — and where do the 9 reported failures concentrate?

## Summary
The Opus bot uses a dual-path message router: deterministic pre-handlers intercept high-confidence Arabic Discord operations (permissions, voice disconnect, nickname, conversation) before the AI, while a multi-provider AI tool loop handles everything else. Entity continuity flows through two parallel systems — `EntityRegistry` (guild-level, 15min TTL) and `MemoryManager` (channel-level, 30min TTL). The 9 reported failures concentrate at: entity TTL asymmetry causing cross-channel reference failure (Problem 1), compound planner regex fragility (Problem 2), split prompt sources for language (Problem 3), name ambiguity in target resolution (Problem 4), all-provider failure collapse to one error message (Problem 5), regex-based tool selection gaps (Problem 6), permission path divergence between direct/sweep/AI (Problem 7), conversational personality sterilization (Problem 8), and implicit reference resolution depending on TTL-fresh entities (Problem 9). The root architectural weakness is the deterministic-to-AI handoff boundary: when deterministic handlers cannot fully process a request, the AI path lacks the session entity context and permission knowledge needed to complete it correctly.

## Detailed Findings

### 1. Message Routing Cascade
- Message entry at `src/index.ts:1298-1313` detects `!opus` or bot mention
- `getConversationReply()` short-circuits small-talk at `src/services/conversation.ts:41-63`
- Deterministic voice handler at `src/index.ts:707-747` intercepts disconnect/limit requests
- Direct Arabic permission handler at `src/index.ts:1389-1419` processes `buildArabicPermissionOperations()` output
- Explicit target resolution via `resolveExplicitToolTargets()` at `src/services/toolTargeting.ts:89-173` runs before AI
- AI request path starts at `src/index.ts:1445` with system prompt assembly at `src/index.ts:1518-1525`
- The routing cascade order determines which problem's canonical example gets intercepted (or fails to intercept)

### 2. Entity Lifecycle and TTL Asymmetry
- `createChannels()` returns `createdEntities` at `src/utils/discordTools.ts:140-219`
- `EntityRegistry.registerToolResult()` at `src/intelligence/entity_registry.ts:143-181` registers guild-level entities
- `MemoryManager.rememberEntities()` at `src/intelligence/memory_manager.ts:314-349` converts to session entities and updates `last_channel_id`
- `buildEntityContext()` at `src/intelligence/memory_manager.ts:388-401` injects `[SESSION_ENTITIES]` into AI prompt
- Follow-up resolution of "الروم" at `src/services/toolTargeting.ts:157-162` checks session entities first, then `EntityRegistry.getLatest()` fallback
- **TTL gap**: `ENTITY_TTL_MS=15min` (registry) vs `SESSION_ENTITY_TTL=30min` (memory) at `src/intelligence/entity_registry.ts:36` and `src/intelligence/memory_manager.ts:43`
- Creates 15-minute window (t=15 to t=30) where registry has dropped entity but session memory still has it
- Cross-channel references fail after 15 minutes (session entities are channel-scoped, registry fallback gone)
- Same-channel references fail after 30 minutes

### 3. Compound Request Paths
- Route 1: Deterministic `planCompoundDiscordRequest()` at `src/intelligence/compound_planner.ts:23-164` — regex-based, limited to `اسمه/اسمها/باسم` name markers
- Route 2: AI-structured `WorkflowEngine.parseFromAIResponse()` at `src/intelligence/workflow_engine.ts:15-45` — LLM produces JSON with `dependsOn` references
- Route 3: Multi-tool AI loop at `src/index.ts:1648-1760` — independent tool calls, no `dependsOn` resolution
- `$create_category.channelId` references resolved by `WorkflowEngine.resolveReferences()` at `src/intelligence/workflow_engine.ts:96-115`
- `applyArabicPermissionsToToolArgs()` at `src/intelligence/arabic_nlp.ts:120-160` transforms Arabic permission text into tool arguments
- Problem 2 fails because the compound planner at `compound_planner.ts:54` only links creation+permissions when BOTH are in the SAME message containing `الكل`; separate messages create channels without permissions

### 4. AI Provider Cascade and Error Surface
- Three-tier cascade: OpenCode Zen (Anthropic) → Cerebras (OpenAI-compatible) → Groq fallback at `src/services/ai.ts:813-881`
- `buildAnthropicBody()` at `src/services/ai.ts:72-166` builds Anthropic-format request with `composeSystemPrompt()` at `:70-72`
- `buildCompletionBody()` at `src/services/ai.ts:614-631` builds OpenAI-compatible request with same `composeSystemPrompt()`
- Circuit breakers at `src/services/ai.ts:362-389` track consecutive failures per provider; 3 failures → 90-second cooldown
- All-provider failure throws `AI_TEMPORARY_ERROR_MESSAGE` at `src/services/ai.ts:380` regardless of root cause
- `formatUserError()` at `src/index.ts:2000-2025` passes the error through unchanged — lossy transformation discards diagnostic context
- Problem 5 (complex request) and Problem 6 (simple nickname) produce identical user-facing error because `generateAIResponse()` at `:870-881` collapses all non-auth failures into one string

### 5. Language Consistency
- `ContextEngine.detectLanguage()` at `src/intelligence/context_engine.ts:80-94` classifies Arabic/Latin/mixed via Unicode script ratio
- `getDominantLanguage()` at `src/intelligence/context_engine.ts:84-94` defaults to Arabic for any non-English input
- System prompt assembled at `src/index.ts:1518-1525` combines `ContextEngine.buildSystemPrompt()`, `buildEntityContext()`, `buildUserPreferenceContext()`, `SkillRegistry.buildSkillManifestForAI()`
- `composeSystemPrompt()` at `src/services/ai.ts:70-72` places runtime prompt AFTER static `SYSTEM_PROMPT` under `[RUNTIME_CONTEXT]`
- Static `SYSTEM_PROMPT` at `src/services/aiCatalog.ts:47` says "Reply in the same language" (generic); runtime prompt says "Reply in Arabic" (specific)
- Problem 3 (English response to Arabic) occurs because models weight earlier prompt sections more heavily; English-first static prompt can override Arabic runtime directive
- `preferred_language=unknown` for new users at `src/intelligence/memory_manager.ts:417-424` provides no useful signal

### 6. Permission Execution Divergence
- Path 1: Direct `buildArabicPermissionOperations()` at `src/intelligence/arabic_nlp.ts:164-204` — single channel, @everyone + optional exception role
- Path 2: `sweep_permission_overwrites` at `src/utils/discordTools.ts:554-641` — multi-channel, per-role discovery, requires exact "منشن+كاتقوري" phrasing at `src/index.ts:765-768`
- Path 3: AI-generated `edit_permissions` at `src/services/aiCatalog.ts:245` — defaults to @everyone via `applyArabicPermissionsToToolArgs()` at `arabic_nlp.ts:143-144`
- All three paths converge on `editPermissions()` at `src/utils/discordTools.ts:442-493` which calls `channel.permissionOverwrites.edit()`
- Problem 7 fails because the sweep path (the ONLY path handling per-role overwrites) requires the exact regex at `src/index.ts:765-768` which demands "منشن/mention" + "كاتقوري/كل الرومات"
- The direct Arabic handler at `src/index.ts:1389-1419` only processes @everyone + exception role pairs; it does not scan for individual roles

### 7. Conversational Personality
- `getConversationReply()` at `src/services/conversation.ts:41-63` provides warm Arabic-only fixed replies for small-talk
- `SYSTEM_PROMPT` at `src/services/aiCatalog.ts:45-121` has strong Arabic personality rules
- `ContextEngine.buildSystemPrompt()` at `src/intelligence/context_engine.ts:96-123` overrides with shorter language directive
- Problem 8 (robotic responses) occurs when `hasActionIntent()` at `src/services/conversation.ts:36-40` detects an ACTION_TERM, sending the message to the AI path where formal tone overrides warm intent
- `buildToolExecutionReply()` at `src/index.ts:1678-1748` adds deterministic structure that further sterilizes AI replies

### 8. Skill Discovery and Execution
- `SkillRegistry.loadDirectory()` at `src/skills/skill_registry.ts:111-127` discovers `.skill.ts` files in `src/skills/`
- `buildSkillManifestForAI()` at `src/skills/skill_registry.ts:144-151` emits only category→ID lines, no schemas
- `execute_skill` tool at `src/services/aiCatalog.ts:257` takes `{skillId, args}` with `additionalProperties: true`
- Permission gate at `src/index.ts:206-209`: empty `requiredPermissions` denies non-admins (`.some()` over empty array returns false)
- Tool adapter at `src/index.ts:529-536` routes through `executeToolWithAudit()` for full safety pipeline
- `KnowledgeSkillLoader` at `src/intelligence/knowledge_skill_loader.ts:71-160` loads `.md` files from project-root `skills/` separately
- AI has no access to skill schemas, descriptions, or argument requirements — must infer from category→ID listing alone

### 9. Missing-Intent Verification
- `detectAllIntents()` at `src/services/intentVerifier.ts:68-76` iterates ALL `INTENT_PATTERNS` (unlike `detectArabicIntent()` which returns first match)
- `findMissingIntents()` at `src/services/intentVerifier.ts:86-103` compares detected intents against executed tool names
- `buildMissingIntentPrompt()` at `src/services/intentVerifier.ts:108-115` injects corrective prompt into secondary AI call at `src/index.ts:1841-1857`
- SET_PERMISSIONS patterns at `src/intelligence/arabic_nlp.ts:16-19` require "صلاحيات/برمشن" or "امنع...يدخل"
- Natural phrasing like "ابي البرمشن حقة الكل يشوف الروم" does match (contains "برمشن"), but "محد يقدر يدخله" without explicit permission keywords does NOT
- The verification loop can only catch intents the NLP patterns recognize — architectural limitation

## Architecture Insights
- The dual-path router (deterministic → AI) is the core architectural pattern; most failures concentrate at the handoff boundary
- Entity continuity is intentionally dual-layered but TTL-asymmetric, creating staggered failure modes by channel scope
- Permission handling has three independent paths with different granularity; only the sweep path handles per-role operations but requires exact phrasing
- The AI provider cascade collapses all failures into one user-facing string, making diagnosis impossible from user reports alone
- Skills are discoverable but not explainable — the manifest gives IDs without schemas, forcing the AI to guess arguments
- The intent verification loop is bounded by NLP pattern coverage — natural Arabic phrasing that doesn't match hardcoded regex bypasses the safety net entirely

- TTL alignment between EntityRegistry and MemoryManager would eliminate the 15-minute cross-channel reference gap

- Skill manifest should include at minimum description + required args schema for the AI to use skills reliably
